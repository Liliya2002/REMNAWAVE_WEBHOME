# Hosting Order Page Integration README

План доработок текущей страницы "Заказать Хостинг" в админке и подключение к внешнему серверу каталога.

## 1. Что уже сделано

- Добавлен маршрут админки: `/admin/hosting-order`.
- Добавлен backend модуль: `/api/admin/hosting`.
- Реализованы endpoints:
  - `GET /api/admin/hosting/catalog`
  - `POST /api/admin/hosting/sync`
- Добавлен локальный кэш каталога в БД.
- Добавлены логи синхронизации.

## 2. Что нужно доделать в текущем проекте

## 2.1 UX и фильтры
- Фильтры на странице: локация, provider, CPU, RAM, price range, in-stock only.
- Сортировки: цена, CPU, RAM, дата обновления.
- Пагинация и виртуализация списка.

## 2.2 Обновление каталога
- Кнопка "Синхронизация по расписанию" (вкл/выкл) в админке.
- Отдельный endpoint для просмотра истории sync logs.
- Карточка "последняя ошибка синхронизации".

## 2.3 Подготовка заказа (без фактической покупки)
- Кнопка "Подготовить заказ" у оффера.
- Создание локальной заявки в таблице `hosting_order_requests`.
- Статусы заявки: `draft`, `queued`, `sent`, `failed`.
- Привязка заявки к администратору и заметкам.

## 2.4 Безопасность
- Отдельный rate limit для `/api/admin/hosting/*`.
- Аудит лог действий: кто и когда запускал sync.
- Маскирование чувствительных данных в ошибках.

## 2.5 Наблюдаемость
- Логи по времени синхронизации.
- Число полученных и измененных записей.
- Метрики по ошибкам подключения к источнику.

## 3. Что нужно для подключения внешнего сервера

## 3.1 На стороне внешнего сервера
- Должен быть доступен endpoint каталога по HTTPS.
- Поддержка auth токена.
- Стабильный JSON контракт офферов.

## 3.2 На стороне текущего проекта
- Настроить env параметры:
  - `HOSTING_CATALOG_URL`
  - `HOSTING_CATALOG_TOKEN`
- Убедиться, что backend может достучаться до внешнего URL.
- Выполнить ручную синхронизацию и проверить кэш.

## 3.3 Тест-кейсы интеграции
- Источник недоступен -> страница не падает, показывает ошибку + старый кэш.
- Источник пуст -> кэш очищается/обновляется корректно.
- Источник отдает обновленные цены -> данные в UI меняются после sync.
- Неверный токен -> понятная ошибка авторизации.

## 4. Предложенная схема новых таблиц (этап следующего шага)

### 4.1 hosting_order_requests
- `id`
- `offer_key`
- `requested_by_user_id`
- `status`
- `request_payload` jsonb
- `external_order_id`
- `error_message`
- `created_at`
- `updated_at`

### 4.2 hosting_provider_events
- `id`
- `order_request_id`
- `event_type`
- `event_payload` jsonb
- `created_at`

## 5. Как будет работать полный цикл в будущем

1. Админ открывает "Заказать Хостинг".
2. Видит кэш каталога из БД.
3. Нажимает "Синхронизировать" для актуализации.
4. Выбирает оффер и создает заявку.
5. Внешний проект обрабатывает заказ и возвращает статус.
6. При статусе `ready` серверные данные переносятся в `vps_servers`.
7. Сервер появляется в "Управление VPS" для дальнейшей настройки и установки ноды.

## 7. Важные решения заранее

- Единый canonical `offer_key` обязателен.
- Контракт API должен быть версионирован (`/v1/...`).
- Заказ VPS должен быть асинхронным (очередь/статусы), а не "синхронным ожиданием".
- Всегда хранить локальный кэш в основном проекте для отказоустойчивости.

## 8. Минимум для production-ready интеграции

- Надежная auth схема между проектами.
- Retry + timeout на внешние запросы.
- Логи sync с историей минимум 30 дней.
- Backup базы каталога.
- Документированный API контракт и версионирование.
